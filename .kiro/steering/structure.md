# Project Structure

## Directory Organization

```
.
├── README.md                           # Project overview and quick start
├── PCS_CONTEXT.md                      # Detailed PCS knowledge base
├── .kiro/
│   ├── steering/                       # Kiro steering files
│   │   ├── product.md                  # Product overview (always included)
│   │   ├── tech.md                     # Technology stack (always included)
│   │   ├── structure.md                # This file (always included)
│   │   ├── pcs-rules.md                # CloudFormation template rules
│   │   └── pcs-interactive-rules.md    # Interactive agent rules
│   ├── agents/                         # Kiro CLI agent configurations
│   │   ├── pcs-interactive-specialist.json  # Interactive CLI-based agent
│   │   └── pcs-cleanup-agent.json      # Cluster cleanup agent
│   └── cli-todo-lists/                 # Agent todo list storage
├── templates/
│   └── merged-pcs-cluster.yaml         # Reference CloudFormation template
└── generated/                          # Agent-generated files and configurations
```

## File Naming Conventions

### Configuration Files
- **CloudFormation**: `pcs-cluster-{timestamp}.yaml`
- **AWS CLI Parameters**: `{resource}-config.json`
- **Generated Scripts**: `{purpose}-{timestamp}.sh`

### Documentation
- **Steering Files**: `{domain}-{purpose}.md`
- **README Files**: `README.md` (uppercase)
- **Context Files**: `{SERVICE}_CONTEXT.md` (uppercase)

### Agent Files
- **Agent Definitions**: `{name}-agent.json`
- **Todo Lists**: Auto-generated by Kiro CLI

## Import and Reference Patterns

### File References in Steering
Use Kiro's file reference syntax for live workspace files:
- Template reference: #[[file:templates/merged-pcs-cluster.yaml]]
- Project overview: #[[file:README.md]]
- Context documentation: #[[file:PCS_CONTEXT.md]]
- Agent configurations: #[[file:.kiro/agents/pcs-interactive-specialist.json]]

### CloudFormation References
- **Parameters**: Use descriptive names with type hints
- **Resources**: PascalCase logical IDs with service prefixes
- **Outputs**: Export important resource IDs for cross-stack references

## Architectural Decisions

### Agent-Based Architecture
- **Interactive Agent**: Step-by-step cluster creation with CLI commands
- **Cleanup Agent**: Safe resource deletion and project reset
- **Specialized Agents**: Domain-specific guidance (security, performance)

### Template Strategy
- **Single Stack**: All resources in one CloudFormation template
- **Modular Design**: Parameterized for different use cases
- **Best Practices**: Based on AWS HPC Recipes and official documentation

### Generated File Management
- **Timestamped Files**: Prevent overwrites and enable versioning
- **Deployment Documentation**: README.md alongside each template
- **Configuration Tracking**: JSON parameter files for reproducibility

## Code Organization Principles

### Separation of Concerns
- **Templates**: Infrastructure definitions only
- **Steering**: Guidance and rules for generation
- **Generated**: Output files from agent interactions
- **Documentation**: User-facing guides and context

### Version Control
- **Include**: Templates, steering files, documentation
- **Exclude**: Generated files (add to .gitignore)
- **Track**: Agent configurations and project structure

### Maintenance
- **Regular Updates**: Keep templates aligned with AWS best practices
- **Documentation Sync**: Update steering files when patterns change
- **Agent Evolution**: Refine agent behavior based on user feedback
